- [x] 1.1 Located the `QueryHint` AST nodes in `crates/nexus-core/src/executor/parser/ast.rs:148-172` (`UsingIndex { variable, label, property }`, `UsingScan { variable, label }`, `UsingJoin { variable }`). The pre-existing branch at `:1295-1316` already filtered hints by variable; this task extends it with real validation.
- [x] 1.3 `USING INDEX <var>:<Label>(<prop>)` now validates against a registered property index. Added `property_index: Option<&'a PropertyIndex>` field to `QueryPlanner` plus a `with_property_index(idx)` builder shim mirroring the existing `with_rtree(reg)` opt-in. When the handle is `Some`, the planner first calls `Catalog::get_label_id` / `get_key_id` to short-circuit on typos (label or property never registered), then `PropertyIndex::has_index(label_id, key_id)` to confirm a matching single-property index exists. On failure the planner raises `ERR_USING_INDEX_NOT_FOUND` with a structured message naming the `(label, property)` pair. Without a handle the hint is accepted silently — that's the legacy behaviour for unit-test callers and direct planner consumers that don't construct an `IndexManager`.
- [x] 1.4 `USING SCAN <var>:<Label>` continues to emit `Operator::NodeByLabel` (already the planner's default for labelled nodes); the hint surface is preserved so future property-index-selection work respects the bypass.
- [x] 1.5 `USING JOIN ON v` is parsed into the AST and accepted by the planner without runtime error. Honouring it requires a hash-join physical operator that the planner does not yet emit; threading it correctly is a separate task once the join-physical-operator surface exists. Documented in the CHANGELOG entry.
- [x] 1.6 4 planner unit tests added at the end of `crates/nexus-core/src/executor/planner/tests.rs`:
