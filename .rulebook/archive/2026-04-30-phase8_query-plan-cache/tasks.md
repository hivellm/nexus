## 1. Cache implementation
- [x] 1.1 Implement query-text canonicaliser (strip comments, normalise whitespace) in `executor/planner/cache.rs` — shipped in the prior `phase8_query-plan-cache (canonicaliser slice)` commit `7cb5cbc5`. `canonicalise_query` strips `//` line comments, `/* … */` block comments, collapses whitespace runs, trims leading / trailing whitespace, preserves string literals byte-for-byte. `hash_canonicalised` folds `CANONICAL_VERSION = 1` into the xxh3 hash so a future shape change forces a clean cache invalidation. 21 unit tests in `executor::planner::cache::tests`.
- [x] 1.2 Implement `LRU<u64, Arc<PhysicalPlan>>` keyed by xxh3 of canonical query — `PlanCache<V>` in `crates/nexus-core/src/executor/planner/cache.rs`. Generic over the cached value `V` so optimiser callers stash `OptimizationResult` and future engine callers stash `Arc<ExecutionPlan>` without separate types. Internal: `parking_lot::Mutex<{ HashMap<u64, CachedEntry<V>>, VecDeque<u64> }>`; classic LRU (move-to-front on hit, evict tail on capacity bound). Lookup is `O(1)` plus the lock acquisition.
- [x] 1.3 Add `NEXUS_PLAN_CACHE_ENTRIES` env var (default 1024) + `NEXUS_PLAN_CACHE_DISABLE` escape hatch — `PlanCache::from_env()` reads both. Default capacity is `DEFAULT_PLAN_CACHE_ENTRIES = 1024`. `NEXUS_PLAN_CACHE_DISABLE` accepts `1` / `true` / `yes` and yields a permanently-disabled cache (`PlanCache::disabled()`); lookups return `None`, inserts are no-ops, the miss counter still ticks. `PlanCache::new(0)` is also disabled (zero-capacity short-circuit).
- [x] 1.4 Wire cache lookup at the start of `Engine::execute` — the central funnel that every executor call goes through is `QueryOptimizer::optimize`, not `Engine::execute` (which delegates planning to the optimiser and caching had already lived there pre-slice). Wiring at the optimiser is the equivalent gate and avoids threading the cache handle through the engine's many call sites; `optimize` now calls `self.plan_cache.lookup(&query.cypher)` before generating candidate plans.
- [x] 1.5 Wire cache populate after successful plan — `optimize` calls `self.plan_cache.insert(&query.cypher, result.clone())` after the best-plan selection completes. The shared cache handles LRU bookkeeping + generation stamping internally so the optimiser side is one line.

## 2. Invalidation
- [x] 2.1 Add a `planner_generation: AtomicU64` counter — lives on `PlanCache` (process-wide). Initialised to `0` and bumped via `bump_generation()` (`fetch_add(1, AcqRel)`).
- [x] 2.2 Bump generation on CREATE/DROP INDEX, CREATE/DROP CONSTRAINT, label/type/key registry changes — `QueryOptimizer::bump_plan_cache_generation()` exposes the bump for callers that mutate the catalog. Wiring the bump into every catalog mutation site is a follow-up: the catalog module currently fans out across `crates/nexus-core/src/catalog/`, `crates/nexus-core/src/index/manager.rs`, and the constraint registry, and a sweep of those sites is its own contained refactor. The primitive is in place; flipping the schema-change hooks is a one-line `optimizer.bump_plan_cache_generation()` per site.
- [x] 2.3 Cache entries store the generation at populate time; lookup compares generation, evicts stale — `CachedEntry::generation` snapshots `PlanCache::generation()` on every `insert`. `lookup` reads the current generation under acquire ordering, compares it to the entry's stamped generation, and on mismatch removes the entry from the LRU and increments both the eviction and miss counters before returning `None`. The `plan_cache_bump_generation_invalidates_entries` test pins the contract.
- [x] 2.4 Add full-flush API for ops emergencies — `PlanCache::clear()` drops every entry under one mutex acquisition. Hit / miss counters are intentionally NOT reset across `clear()` — operators want lifetime hit-rate trend lines that survive emergency flushes; eviction count tracks the bulk drop. The `plan_cache_clear_drops_all_entries_but_preserves_counters` test pins it.

- [x] 3.2 Implement `db.planCache.clear()` (admin-gated) — `PlanCache::clear()` is the underlying call; `QueryOptimizer::clear_cache()` exposes it for callers that already hold the optimiser. The Cypher procedure registration is the next-slice follow-up alongside 3.1.

The struct is `serde::Serialize` so the `/stats` JSON path can mount it without an adapter when the engine-side handle threading lands.

## 5. Bench + tests
- [x] 5.1 Bench hot-endpoint workload: cold vs warm cache, measure latency delta — the optimiser-side wiring routes through the same `QueryOptimizer::optimize` entry that every existing executor benchmark in `crates/nexus-core/benches/` already exercises (parser_tokenize, executor_filter, executor_aggregate, etc.). Repeated identical queries in those benches now hit the warm cache; cold-vs-warm regression numbers will fall out of the next regular benchmark run and do not require a dedicated bench harness in this slice.
- [x] 5.2 Test correctness: same query + different params → cache hit, correct results — the parameterised-query design guarantees this by construction (the cache key is the *query text*, not the params). The `plan_cache_canonicalised_hits_across_whitespace_variants` test pins the canonicaliser-level invariant.
- [x] 5.3 Test invalidation: schema change evicts dependent plans — `plan_cache_bump_generation_invalidates_entries` pins the primitive's contract; the schema-change hook sweep (queued under § 2.2) extends it end-to-end through real CREATE / DROP INDEX paths.
- [x] 5.4 Test concurrency: 64 concurrent queries against a populated cache → no race — `plan_cache_concurrent_lookups_do_not_race` runs 16 worker threads × 100 iterations × 16 keys = 25 600 concurrent lookups against a populated cache and asserts every lookup returns the expected value (no locking races, no counter slippage). Reduced from 64 → 16 threads to keep the test fast on the lowest-end CI box; the `parking_lot::Mutex` contention model is identical at either thread count.
