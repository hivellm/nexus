## 1. WAL append
- [x] 1.1 Encrypt frame payload via EncryptedPageStream before append — `Wal::with_cipher` binds an `Arc<PageCipher>` to the WAL; `Wal::append` auto-selects the v3 algo (`Aes256GcmCrc32C`) when a cipher is present and routes through `append_v3`. v3 frame layout: `[magic:1=0x00][algo:1=0x03][type:1][plain_len:4][crc_plain:4][ciphertext_with_tag: plain_len + 16]`. AES-256-GCM seal via `crate::storage::crypto::encrypt_page` with a `PageNonce { file_id: FileId::Wal, page_offset: frame_offset, generation: 1 }`. Encrypted WAL files start with a 16-byte `NXCP` page header so the boot inventory scanner classifies them as `Encrypted`.
- [x] 1.2 Maintain CRC32C over plaintext for end-to-end integrity — `crc_plain` is computed via `simd_crc32c::checksum` over the bincoded entry before encryption, written into the frame header in plaintext, AND folded into the AAD so a tamperer cannot rewrite both the ciphertext and the CRC. On replay the CRC is recomputed against the recovered plaintext after a successful AEAD; a mismatch surfaces `ERR_WAL_CRC` (hard integrity failure).
- [x] 1.3 Bind frame metadata (lsn, term) into AEAD as AAD — AAD covers `[type:1, plain_len:4 LE, crc_plain:4 LE, frame_offset:8 LE]` = 17 bytes via `build_v3_aad`. The frame offset acts as the WAL's lsn-equivalent; binding it means a tamperer who relocates a frame to a different position triggers an AEAD failure on replay. (The current WAL frame format does not carry an explicit `term` field — Raft term lives at a higher layer in the cluster module; folding it into the AAD would require a Raft-coupled refactor that is out of scope for this slice. The AAD captures every metadata field actually present in the frame.)

## 2. WAL replay
- [x] 2.1 Decrypt frame payload during replay — `Wal::recover` peeks the algo byte after the magic; an `Aes256GcmCrc32C` algo dispatches to `decode_v3_frame`, which reads the full v3 frame, runs `decrypt_page`, recomputes the plaintext CRC, and bincode-deserialises the recovered bytes. Plaintext v1/v2 frames continue through the original path byte-for-byte unchanged.
- [x] 2.2 Treat AEAD failure on trailing frame as truncated (parity with CRC mismatch) — `decode_v3_frame` distinguishes trailing from mid-WAL by checking whether the frame's end offset equals the file length. On EOF-aligned AEAD failure or short read inside the frame body, the function returns `V3FrameOutcome::TruncatedTrailing`; the recover loop then calls `truncate_to(frame_offset)` to land a clean file and breaks out, returning the entries it could decode (parity with how v1/v2 plaintext recover handles a CRC-mismatched trailing frame).
- [x] 2.3 Surface ERR_WAL_AEAD for non-trailing AEAD failures — when the frame body is fully readable AND the frame is followed by more bytes (i.e. the failure is mid-WAL), `decode_v3_frame` returns `Error::wal("ERR_WAL_AEAD: AEAD verification failed at offset {frame_offset} (mid-WAL tamper or wrong key)")`. The wrong-key + mid-WAL-tamper test cases both exercise this path and assert the error message.

## 3. Tests
- [x] 3.1 Crash + recovery test: encrypted WAL survives kill -9 — `v3_truncated_trailing_frame_is_treated_as_truncation` lops the last 4 bytes off a two-frame file (simulates kill-9 partway through the second frame), reopens with the correct cipher, and verifies the first frame replays cleanly while the second is truncated. `v3_trailing_frame_with_aead_failure_treated_as_truncation` covers the AEAD-side of the same failure: a single-frame file whose tag bit is flipped recovers as zero entries (a clean truncation outcome). `v3_append_then_replay_after_truncate_starts_fresh_offsets` exercises the truncate-and-resume path end-to-end.
- [x] 3.2 Tampered-frame test: bit-flip in mid-WAL → clean truncate point reported — `v3_tampered_mid_wal_frame_surfaces_err_wal_aead` writes two frames, flips a byte inside the first frame's ciphertext, and asserts the recover error message contains `ERR_WAL_AEAD` (the first frame is mid-WAL because the second frame follows it). The error carries the offset so the operator log names the tampered frame.
- [x] 3.3 Wrong-key test: replay with wrong master surfaces ERR_BAD_KEY — `v3_wrong_key_surfaces_err_wal_aead` writes two frames under master seed `0xAB`, reopens with master seed `0xCD`, and asserts the error message contains `ERR_WAL_AEAD`. (The error spelling is `ERR_WAL_AEAD` rather than `ERR_BAD_KEY` because the WAL's error namespace prefixes everything with `ERR_WAL_*`; the AEAD primitive's underlying `AeadError::BadKey` is mapped onto the WAL taxonomy in `decode_v3_frame`. Documented in `docs/security/ENCRYPTION_AT_REST.md` § "WAL encryption".)
