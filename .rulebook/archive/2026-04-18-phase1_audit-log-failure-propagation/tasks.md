## 1. Implementation
- [x] 1.1 Policy decided with owner 2026-04-18: **fail-open with metric + tracing::error**, documented in `docs/SECURITY_AUDIT.md §5 "Audit-log failure policy (fail-open)"` and `docs/AUTHENTICATION.md` "Audit-log failure handling" section. Rationale: all 4 middleware call sites are already failure paths (401/429/500); converting them to 500 on audit-sink failure hands an attacker a lever to mass-reject legitimate traffic by causing disk/IO pressure.
- [x] 1.2 Replaced the 4 `let _ = audit_logger.log_authentication_failed(...)` in `nexus-core/src/auth/middleware.rs` (missing API key, invalid API key, verify_api_key error, rate-limit exceeded) with `if let Err(e) = ... { record_audit_log_failure("<tag>", &e); }`. The tag lets operators correlate counter movement with the originating call site.
- [x] 1.3 Grepped `let _ = .*audit` across the repo — no remaining sites in `nexus-core/src/auth/` after step 1.2. Found 4 companion sites in `nexus-server/src/api/cypher/execute.rs` (SET-property and SET-label success/failure on the write path) covering the same "audit event dropped silently" pattern; applied identical fail-open treatment via `nexus_core::auth::record_audit_log_failure("set_property_success" | "set_property_failure" | "set_label_success" | "set_label_failure", &e)`.
- [x] 1.4 Added `nexus_audit_log_failures_total` counter exported at `GET /prometheus`. Sourced directly from the `nexus_core::auth::middleware::AUDIT_LOG_FAILURES` process-global `AtomicU64` via `audit_log_failures_total()` getter so the metric surfaces bumps from ANY call site — auth middleware or write path — without per-site plumbing. HELP text explicitly reminds operators to alarm when the counter moves.
- [x] 1.5 `cargo +nightly clippy -p nexus-core -p nexus-server -- -D warnings` — zero warnings after the changes.
