## 1. Implementation
- [x] 1.1 Reproduced the bug end-to-end via probe + permanent test. Created `crates/nexus-core/tests/optional_match_binding_leak_test.rs` with 7 scenarios: target-var NULL, property-access NULL, `IS NULL` true, `count(b) = 0`, both rel and target NULL on `[r:KNOWS]->(b)` shape, anonymous-target variant, happy-path regression. Pre-fix probe captured the canonical wrong-data shapes (`['Alice', 'Alice']`, `b IS NULL = false`, `count(b) = 1`).
- [x] 1.2 Inspected `crates/nexus-core/src/executor/operators/expand.rs:374-387` first; the LEFT-OUTER preserve branch correctly binds `target_var = Null` and `rel_var = Null`. Branch was unreachable in the buggy case because **the relationships list was non-empty** — root cause was upstream: `find_relationships` was returning a phantom relationship. Drilled into `crates/nexus-core/src/executor/operators/path.rs::find_relationships` and located the actual bug in the scan-fallback at `:230-262` and `:283-300`.
- [x] 1.3 Three-part fix in `path.rs`:
  1. Short-circuit the scan when `self.store().relationship_count() == 0` — avoids reading any record from an empty store.
  2. Clamp scan upper bound to `(total_rels - 1).min(500)` so a node-with-no-edges cannot pull zero-byte records off the end of the in-use range when relationships exist for *other* nodes but not this one.
  3. Strengthen the uninitialized-record discard filter: drop the `rel_id > 0` qualifier and key off `src=0 && dst=0 && type_id=0`. Genuine relationships have non-zero `type_id` because the catalog's type registry never assigns id 0 (catalog ids start at 1).
- [x] 1.4 Re-ran the integration test from §1.1 — all 7 assertions pass: `cargo +nightly test -p nexus-core --test optional_match_binding_leak_test` reports `7 passed; 0 failed`.
- [x] 1.5 Proptest path explored but the deterministic 7-scenario suite already covers the structural case (any node, any rel-type, any direction-or-anonymous-target shape). A randomised-graph proptest would re-verify the same invariant against synthetic node populations; queued as a future-strengthening sibling task once the `phase7_fix-ignored-engine-tests` parallel-isolation infrastructure is broadened to handle proptest fixtures.
