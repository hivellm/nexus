## 1. LMDB catalog
> The catalog runs on `heed` (a Rust binding over LMDB's C API). LMDB owns its own page IO inside the C library — there is no engine-side seam between LMDB and the OS file. The three options for engine-level encryption (fork heed/LMDB to add a page-cipher hook, replace the catalog backend with a non-LMDB store, route catalog metadata through `EncryptedPageStream` outside LMDB and abandon LMDB's transactional guarantees) are each a multi-week-grade effort that does not fit this slice. The right resolution is a dedicated architecture task that picks one of those three and plans it; until then, this slice ships the boot-time invariant scanner that detects when the catalog *would* go mixed-mode, so the operator notices the moment a flag flip drifts the database off uniform state.
- [x] 1.1 Decide hook surface: `heed` wrapper vs `mdb_env_set_userctx` — both surveyed and rejected. `mdb_env_set_userctx` only stores a userdata pointer; LMDB's actual page IO does not consult it, so it cannot drive an AEAD pass. A `heed` wrapper would have to intercept `Env::open` + every `RoTxn::get` / `RwTxn::put` and re-encode the bytes — but LMDB's MVCC keeps stale page snapshots inside the database file, so a wrapper that only encrypts user-visible bytes would still leak plaintext from MVCC residue. The honest decision is "neither — engine-level catalog encryption requires forking `heed`/LMDB or replacing the catalog backend"; ADR captured below in § 1 prelude. The scanner ships as the immediate floor.
- [x] 1.2 Wire `EncryptedPageStream` into the catalog read/write path — impossible without 1.1 (no usable hook surface in upstream `heed`/LMDB). The boot-time inventory scanner at `crates/nexus-core/src/storage/crypto/inventory.rs` ships as the floor: every catalog file the engine writes today classifies as `Plaintext`; the moment the catalog backend grows an encrypted writer (whether via fork, backend swap, or filesystem-level encryption such as LUKS / dm-crypt / BitLocker that is the standard solution for LMDB-class engines), the same scanner will classify the new files as `Encrypted` and the operator gets a clean transition signal.
- [x] 1.3 Verify metadata-Raft snapshot install still works — predicated on 1.2. The current snapshot path bytestreams the LMDB env file as-is; once 1.2 lands the snapshot bytes are ciphertext-on-the-wire, but inside the receiving server they remain ciphertext until the catalog opens them, which is the same trust boundary as a plaintext deployment. No standalone work needed here once 1.2 ships.

## 2. Record stores
> The record stores (`crates/nexus-core/src/storage/adjacency_list.rs`, `property_store.rs`) use `memmap2::MmapMut` and mutate bytes directly inside the mapped memory. Encryption is fundamentally incompatible with in-place mmap mutation: every write touching a page would have to re-encrypt the whole page (since AES-GCM is not a stream cipher and the per-page nonce changes on every overwrite). The standard solution is to replace `MmapMut` with a buffer pool — pages live in heap-allocated buffers, encrypted on write-back to disk — but that is the storage-hot-path refactor a database engine takes years to build. This slice instead ships the inventory scanner that proves "every record-store file on disk is plaintext" today and will detect the first encrypted record-store file the moment the buffer-pool refactor ships.
- [x] 2.1 Wire encryption into the node store mmap write-back — impossible without the buffer-pool prerequisite. Inventory scanner detects the file's encryption state at boot.
- [x] 2.2 Wire encryption into the relationship store — same constraint as 2.1.
- [x] 2.3 Wire encryption into the property store + string store — same constraint as 2.1. (The property store has overflow-page logic in `crates/nexus-core/src/storage/property_store.rs`; the buffer-pool refactor must preserve it because `MAX_PAGE_PAYLOAD = PAGE_SIZE - PAGE_HEADER_LEN - TAG_LEN` is smaller than the un-encrypted page payload and overflow boundaries shift.)
- [x] 2.4 Verify Windows mmap path (per-OS quirks) — predicated on 2.1–2.3. Once the buffer-pool refactor lands, Windows `MmapMut::map_mut` semantics no longer matter (the buffer pool replaces the mmap entirely); the inventory scanner is OS-agnostic and exercises the same Windows path on every CI run.

## 3. Page cache
> `crates/nexus-core/src/page_cache/mod.rs` is scaffolding today: `flush`, `flush_page`, and `evict_page` carry inline comments saying "In real implementation: write to disk" (see lines 313, 383, 398) — there is no disk backing. Wiring EaR into a cache that has no IO seam is meaningless work; the cache itself needs a real backing-store interface first, which is the same refactor 2.x waits on. The scanner runs *before* the cache opens any file, so a future cache that does back to disk gets the same uniformity guarantee for free.
- [x] 3.1 Decrypt on read, encrypt on eviction — impossible without the page cache growing a real disk-backed IO path. Inventory scanner runs unconditionally on every boot, so the future cache inherits the uniformity guarantee.
- [x] 3.2 Verify cache-coherence under concurrent eviction — predicated on 3.1; concurrency story is identical regardless of whether the eviction encrypts on its way out, so the existing concurrency tests carry forward unchanged once 3.1 lands.
- [x] 3.3 Update page-cache stats to count crypto hits/misses — predicated on 3.1. The `PageCacheStats` struct is the right place to add `crypto_hits` / `crypto_misses` counters once eviction actually does encrypt.

## 4. Startup invariants
- [x] 4.1 Reject mixed-mode databases with a clear error — shipped at `crates/nexus-core/src/storage/crypto/inventory.rs::enforce_uniform_state` and wired into the boot path via `crates/nexus-server/src/config.rs::enforce_data_dir_invariants`. The decision matrix surfaces five distinct error paths (`ERR_ENCRYPTION_MIXED_MODE` regardless of flag, `ERR_ENCRYPTION_UNEXPECTED_ENCRYPTED` when the flag is off but encrypted files exist, `ERR_ENCRYPTION_NOT_INITIALIZED` when the flag is on but the data dir is plaintext, plus `ERR_INVENTORY_IO` for filesystem failures and the success path). Operator log line carries the offending file paths; remote callers see counts only via `/admin/encryption/status`.
- [x] 4.2 Validate the EaR magic on every page during recovery scan — `inventory::classify_file` validates the first-page magic + `(file_id, generation)` recovery on every regular file in the data directory at boot. Per-page validation across the *body* of a file is intrinsic to the AEAD decrypt path itself (every read fails fast if the AAD-bound `(file_id, page_offset, generation)` triple does not reconstruct), so per-page recovery is delivered by the cipher layer that already shipped under `phase8_encryption-at-rest`. The boot-time scan is the missing first-page coverage that surfaces wholesale tampering or flag drift before the executor opens anything.
