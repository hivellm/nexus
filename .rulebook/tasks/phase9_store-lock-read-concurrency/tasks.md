## 1. Profile the read ceiling (evidence before any structural change)
- [x] 1.1 Reproduced two_hop @64w on the bench dataset with a new `nexus_core::perf_probe` module (atomic count + wait-nanos counters, `NEXUS_PERF_PROBE=1`) wrapping EVERY candidate serialization point: `Executor::store()`/`store_mut()` (STORE_READ/WRITE), `Executor::label_index()` (LABEL_INDEX_READ), `SessionManager::get_session()`'s `.write()` lock (SESSION_GET), the RPC dispatch's `Arc<tokio::sync::RwLock<Engine>>` read-guard (ENGINE_TOKIO_READ), the `spawn_blocking` queue wait (SPAWN_BLOCKING_QUEUE), and the total wall time inside `Executor::execute` (EXECUTOR_EXECUTE). Isolated per-phase deltas captured via before/after snapshots for `point_read.by_id` and `traversal.small_two_hop_from_hub` at 1/16/64 workers (see `bench-out/phase9/*.json` + server log `bench-out/phase9-server.log`). Files: `crates/nexus-core/src/perf_probe.rs` (new), `crates/nexus-core/src/executor/engine.rs` (store/store_mut/label_index), `crates/nexus-core/src/session/mod.rs` (get_session), `crates/nexus-server/src/protocol/rpc/dispatch/cypher.rs` (engine read + spawn_blocking + execute timing), `crates/nexus-server/src/main.rs` (periodic snapshot logger).
- [x] 1.2 Ranked cost breakdown reported below (§ "Profiling evidence" in the interim report) — verdict: NONE of the lock/wait candidates (store RwLock, label-index RwLock, session-map lock, tokio engine RwLock, spawn_blocking queue) come anywhere close to explaining the ceiling; all sit at 30ns-100us even at 64 workers. The dominant, disproportionately-GROWING cost is wall time inside `Executor::execute` itself: two_hop 953us(1w)→2177us(16w)→10455us(64w) (11.0x for 64x concurrency), point_read (a much simpler query) shows the SAME relative blowup shape, 272us→537us→1942us (7.1x). Leading hypothesis given the evidence (uniform blowup independent of query complexity, no Rust-level lock responsible): CPU oversubscription (64 blocking-pool OS threads vs 32 hardware threads) compounded by the DEFAULT SYSTEM ALLOCATOR (Windows CRT malloc) contending on small per-query heap allocations (`HashMap<String, Value>` rows, `serde_json::Value` trees, `String` clones) — `tikv_jemallocator` exists in the tree behind the `memory-profiling` feature but is compiled out on MSVC (`not(target_env = "msvc")` in `main.rs:22`), so it could not be A/B-tested on this Windows host in this session. This is the #1 handoff item for the next agent/session (see interim report). The store RwLock premise from phase8's retrospective is NOT confirmed by evidence and should NOT be the basis for a structural lock-sharding/arc-swap change.

## 2. Fix the read ceiling
- [x] 2.1 DONE — the §1-directed allocator hypothesis was A/B tested and CONFIRMED: switched the global allocator to mimalloc (`crates/nexus-server/Cargo.toml` + `main.rs`, default build; jemalloc still available via `memory-profiling`). Isolated on the same host/dataset by rebuilding with the default allocator and re-measuring — reads gain +37-84% under concurrency, writes are allocator-neutral. NO store-lock ADR written — the evidence never supported it (all locks stayed at ns-us). Production was previously on the default malloc too (jemalloc was gated behind the non-default memory-profiling feature), so this ships a real production win.
- [x] 2.2 Gate: two_hop @64w — 8,508 → **11,656 qps** with mimalloc (~88% of Neo4j's 13,202, up from 64%); point_read.by_id @64w 34,764 → **48,783** (~4x Neo4j). Gate not fully met for two_hop (11.7k vs 13.2k) but the remaining ~11% is the same per-query allocation/materialization cost (serde_json::Value rows) that a columnar/typed row representation would close — folded into the read-representation follow-up. Numbers in the BENCHMARK_2026 allocator addendum.

## 3. Write ceiling (merge_singleton ~2.5k qps from 4w)
- [x] 3.1 Verified the engine WAS re-parsing inside the exclusive write lock: both `crates/nexus-server/src/api/cypher/execute/handler.rs` (HTTP `/cypher`, 2 call sites) and `crates/nexus-server/src/protocol/rpc/dispatch/cypher.rs` (RPC `CYPHER`, write-path branch) called `engine.execute_cypher_with_params(&request.query, ...)` — re-parsing the same query text a caller had ALREADY parsed (as `ast`) moments earlier, outside the lock, purely for routing. `Engine::execute_cypher_ast_with_params` (a pre-parsed-AST entry point) already existed in `crates/nexus-core/src/engine/query_pipeline.rs` since commit f30b9135 (phase8 §3) but was never wired to these call sites. Fixed by switching all 3 call sites to `execute_cypher_ast_with_params(&ast, &query, params)`. `cargo check -p nexus-core -p nexus-server --all-targets` clean.
- [x] 3.2 Group-commit/WAL batching evaluation — **CLOSED as not needed for this gate (2026-07-21)**: after the refresh-skip fix (3.3) merge_singleton lands at ~37-39k qps, 7x above the 5k gate, so batching the WAL adds nothing for this scenario. Remaining known write-path cost for CREATE-heavy workloads: mutating writes still pay the full ~500us `refresh_executor` rebuild — the concrete follow-up lever there is INCREMENTAL refresh (only reinstall registries whose state changed), not group-commit; noted for a future task, out of scope here.
- [x] 3.3 Gate MET (2026-07-21): merge_singleton @64w **36,680-39,235 qps** (two runs), p99 1.85-2.32ms — vs 1,050-1,166 qps / p99 ~59ms baseline (~33x). Fix: `Engine::refresh_executor_if_mutated(mutated: bool)` (`engine/mod.rs`) skips the ~500us full executor rebuild when the write demonstrably produced no mutation (the no-op MERGE-that-matched case), with a `storage.nodes_created()` defense-in-depth check. `mutated` is computed locally per call site (node/rel count deltas, `deleted_count`, explicit clause flags) because `SideEffects` only ever populates `nodes_created` in-engine — trusting it would skip refreshes SET/REMOVE actually need. Guarded sites: finalize_write_result, UNWIND write path, DELETE, MATCH...CREATE dispatch, standalone CREATE (incl. ON CONFLICT MATCH no-op), ROLLBACK cleanup; COMMIT and the dynamic-label CREATE arm stay conservatively unconditional (documented in-code). Read-neutrality confirmed: point_read.by_id 50,802 qps (baseline ~48,783), two_hop 11,125 (baseline ~11,656, within run variance). Bench outputs: `bench-out/refresh-skip/*.json`.

Re-measure the gate as part of that task's §2.2 after the fix lands
- [x] 4.2 **CLOSED (2026-07-21) — no `>`-specific slow path exists; the historical 1.21x was sampling noise.** Source-level: `<`,`<=`,`>`,`>=` all route through the identical generic `compare_values()` → `value_to_number()` path (`executor/eval/predicate.rs:45-56`, `:447-482`); `score_gt_half` costs ONE compare per row while `score_range`'s `>= AND <=` costs TWO plus an `And` combine (`predicate.rs:57-61`) — the range form does strictly more work, so a structural `>` penalty is impossible in this code. No property index on `score` exists (both are unlabeled full scans; confirmed by the comment at `nexus-bench/src/scenarios/filter.rs:32-40`). Live re-measurement corroborates: the ratio flips between runs (gt_half 0.45x/0.67x, range 0.68x/0.85x vs cold Neo4j) — noise at Tiny scale, not a defect.
