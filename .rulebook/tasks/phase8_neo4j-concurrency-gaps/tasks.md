## 1. Aggregation-read concurrency collapse (count_all: 2.9k qps / p99 124ms @64w vs Neo4j 13k)
- [x] 1.1 Reproduced at 1/4/16/64w on the live bench dataset (2584-2699 live nodes). Root-caused via a throwaway `parse_and_plan` diagnostic (not committed): `MATCH (n) RETURN count(n) AS c` plans to `[AllNodesScan, Project{n->n}, Aggregate]` — the identity `Project` the planner inserts for the RETURN item disqualified `Executor::try_short_circuit_count_cross_product` (`crates/nexus-core/src/executor/dispatch.rs`), so this exact aliased shape ALWAYS fell through to the full per-node JSON materialisation path (`execute_all_nodes_scan`/`read_node_as_value`: label lookup + property-chain load per node), not the header-only count the short-circuit exists for. Layered on top: `RecordStore::read_node` (`crates/nexus-core/src/storage/record_store_ops.rs`) re-validates `prop_ptr` against `property_store` on every call (2nd lock), and both `count_live_nodes_all`/`count_live_nodes_for_label` re-acquired the `nodes_mmap` lock once per node — O(n) lock acquisitions per query, the actual 16w-\>64w collapse source once the short-circuit gap above was closed.
- [x] 1.2 Fix (3 parts, `crates/nexus-core/src/executor/dispatch.rs` + `crates/nexus-core/src/storage/record_store_ops.rs`): (a) `try_short_circuit_count_cross_product` now skips a trailing identity `Project` (every item is `Variable(v) -> v`, provably a no-op) between the scan(s) and the `Aggregate`; (b) added `RecordStore::read_all_node_headers()` — one `nodes_mmap` lock acquisition, bulk `bytemuck` cast to `&[NodeRecord]`, bounds by `node_count()` (not the pre-grown mmap length, to avoid counting zeroed trailing slots as "not deleted"); (c) `count_live_nodes_all`/`count_live_nodes_for_label` now call it once and index/filter in memory instead of re-locking per node. Correctness: `cargo +nightly test -p nexus-core --lib` 2414 passed / 0 failed / 10 ignored (incl. `count_over_label_cartesian_product_matches_catalog_product`, all `aggregate::tests::*`), clippy clean.
- [x] 1.3 Gate MET — re-ran `concurrent_sweep` (same bench-data dir, 2584 nodes, 15s/2s per level): count_all @64w **62,653 qps** (target ≥8.6k, 7.3x over), p99 **2.25ms** (target <15ms). Full curve: 1w 3,648 / 4w 12,433 / 16w 42,324 / 64w 62,653 qps — near-linear scaling replacing the prior 546/1,681/2,508/2,876 flatline. Raw: `bench-out/phase8/count-all-item1-final.{json,md}`.

## 2. Traversal scaling ceiling (two_hop: 8.1k qps @64w vs Neo4j 13.2k)
- [x] 2.1 Profiled via curl timing on the live bench dataset (50-node `:P` hub). Root cause is NOT storage/page-cache locking or `spawn_blocking` pool sizing — single-request latency is already ~1ms flat regardless of concurrency, so this is pure lock-acquisition-count contention, same family as item 1: `MATCH (:P {id: 0})...` has no property index on `:P(id)` (confirmed via the engine's own `Nexus.Performance.UnindexedPropertyAccess` notification), so the planner falls back to `NodeByLabel` + `Filter` — `execute_node_by_label` (`crates/nexus-core/src/executor/operators/scan.rs`) calls `self.read_node_as_value(node_id)` once per `:P` node (50 nodes), and each call independently acquires the `ExecutorShared`-level `store` `parking_lot::RwLock` (`Executor::store()`, `crates/nexus-core/src/executor/engine.rs:312`) — 50 acquisitions of that single global lock per query, on top of the nested per-node `nodes_mmap`/`property_store` locks `read_node_as_value` itself takes.
- [x] 2.2 Fix applied (bounded scope — see note below): (a) `Executor::read_node_as_value` (`crates/nexus-core/src/executor/operators/path.rs`) now holds ONE `self.store()` guard for both its header read and property load (was 2 acquisitions, now 1); (b) `RecordStore::load_node_properties_with_ptr` (`crates/nexus-core/src/storage/record_store_ops.rs`) — new entry point that accepts a caller-already-has `prop_ptr` instead of `load_node_properties(node_id)` internally re-reading the node record (removed a redundant `nodes_mmap` + `property_store` round trip per node); `load_node_properties_inner` also now acquires its `property_store` guard ONCE per call instead of up to 3 times (entity-info check, offset load, reverse-index fallback all shared one guard). Correctness: `cargo +nightly test -p nexus-core --lib` 2414/0/10 green, clippy clean. **NOT fully fixed**: the dominant remaining cost is the OUTER `ExecutorShared.store` `parking_lot::RwLock` still being acquired once per node inside `execute_node_by_label`'s loop (via `self.read_node_as_value`), because `read_node_as_value` acquires its own guard internally — hoisting that outer acquisition to once-per-scan (mirroring item 1's `read_all_node_headers` bulk pattern) requires threading an already-held `&RecordStore` through `read_node_as_value` and every one of its ~6 call sites (`scan.rs` x3, `expand.rs`, `quantified_expand.rs`, `path.rs` itself) — a wider, higher-blast-radius refactor than the two applied here, deliberately not attempted in this pass given the risk/time trade-off. Documented as the concrete next step rather than silently dropped.
- [x] 2.2b Deeper refactor DONE (the "not yet pulled" lever from 2.2): `read_node_as_value` now has a `_with_store(&RecordStore, id)` variant, and the ~6 bulk scan call sites (`scan.rs` ×3, `expand.rs`, `quantified_expand.rs`, `path.rs`) acquire ONE `store()` guard per scan loop and call the guarded variant per element — audited each converted loop for nested same-lock acquisition (property loads take the DIFFERENT `property_store` lock, safe). Also threaded through `eval/helpers.rs`. Green: nexus-core 2414/0, clippy clean.
- [ ] 2.3 Gate NOT met after the full refactor — two_hop @64w **8,508 qps** (verified sweep, `bench-out/phase8/two-hop-item2-verified.md`); curve 938 / 2,848 / 6,631 / 8,508 at 1/4/16/64w vs Neo4j 13.2k. Removing the per-node store-lock acquisitions moved 1-16w latency but the 64w ceiling is unchanged: the residual contention is NOT the per-node lock count (now hoisted) but the single global `ExecutorShared.store` `RwLock` itself — 64 concurrent readers each take a fresh read guard per query and contend on the one lock word. Closing this needs a structurally different fix (lock sharding, an `arc-swap` snapshot of the store, or a genuinely lock-free read path over the mmap) — a design-doc-level change, larger than this task's other items. Recorded honestly rather than forced; carried as the headline follow-up.
