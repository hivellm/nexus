# QPP path modes: explicit-keyword routing must disable legacy *m..n lowering
**Source**: manual
**Date**: 2026-04-30
**Related Task**: phase8_quantified-path-patterns-execution
**Tags**: cypher, qpp, path-mode, planner-lowering, bfs-dedup
When threading `QppMode` through the QPP pipeline, the obvious bug is to honour the mode field at the operator level but leave the parser-side `try_lower_to_var_length_rel` unchanged. Lowering replaces the QPP group with a legacy variable-length-relationship pattern, so the operator-level mode field never reaches the executor — the engine silently runs WALK semantics regardless of what the user typed.\n\nFix: any *explicit* mode keyword (including `WALK`) must disable the legacy lowering, even when the body shape would otherwise qualify. Add a `mode_explicit: bool` flag on `QuantifiedGroup` (set by the parser when a mode keyword is consumed) and gate `try_lower_to_var_length_rel` on it. Default-WALK (no keyword) keeps lowering on for backward compatibility.\n\nSecondary trap: BFS wavefront dedup `(node, iteration)` is correct only under WALK. In TRAIL / ACYCLIC / SIMPLE two paths reaching the same `(node, iteration)` via distinct sub-paths have distinct visited sets — collapsing them silently loses extensions. Disable the dedup in non-WALK modes.\n\nSee `crates/nexus-core/src/executor/parser/ast.rs::try_lower_to_var_length_rel` and `crates/nexus-core/src/executor/operators/quantified_expand.rs::execute_quantified_expand`.
